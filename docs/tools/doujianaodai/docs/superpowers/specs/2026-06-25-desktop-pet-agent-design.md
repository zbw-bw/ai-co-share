# 桌面宠物 Agent 设计文档

> 项目名：doujianaodai（逗叽脑袋）
> 日期：2026-06-25
> 状态：MVP 设计

---

## 1. 项目定位

一个基于 Hermes Agent 框架的 Windows 桌面宠物 Agent。在 Hermes 已有的 Agent 能力（Skills、Tools、MCP、Memory、对话）基础上，新增**被动屏幕监控**能力——自动截图、OCR 识别、理解用户正在做什么，将活动以文档形式写入记忆。用户可以随时通过对话询问"今天做了什么"，Agent 基于记忆回答。

### 核心差异点

当前 Agent 产品都是主动触发式，用户不会把一天的所有活动都主动告诉 Agent。本项目通过被动监控填补这个空白，让 Agent 真正了解用户的日常活动。

---

## 2. MVP 范围

### 包含

- 截图 + 本地 OCR + 活动记录（被动监控核心链路）
- 活动分类与标签（基于场景自动打标签，如 #阅读 #写作 #学习 #工作）
- 对话问答（基于活动记忆回答用户问题）
- Hermes Agent 原有能力（Skills、Tools、MCP）

### 不包含（后续迭代）

- 悬浮窗桌面宠物形象和动画（MVP 用简单对话窗口）
- 智能截图频率（屏幕变化检测、停留时间判断）
- 隐私过滤（自动跳过聊天记录）
- 自定义宠物性格（用户自定义提示词/角色）
- 视频标题识别 + 搜索概要
- 看代码 / 写代码场景识别

### MVP 场景

仅支持两种场景：

1. **阅读文章/文档** — 用户在浏览器、PDF 阅读器、文档软件中阅读内容
2. **编写文档** — 用户在 Office、WPS、Notion 等工具中编写内容

### 双 Loop 并行

1. **主动交互 Loop（Hermes 已有）** — 用户发消息时触发，走标准 Agent 流程（理解意图 → 选工具 → 执行 → 回复），可调用 MCP、Skills 等
2. **被动监控 Loop（新增）** — 后台线程持续运行，定时截图 → OCR → 场景分类 → 生成摘要 → 写入 Hermes Memory

两个 Loop 共享 Hermes 的 Memory 系统，主动交互 Loop 可以检索被动监控写入的活动记录。

## 5. 被动监控系统

### 监控流水线

```
定时器(30s) ──► 截图 ──► OCR识别 ──► 场景分类 ──► 活动摘要 ──► 写入记忆
                                       │
                                  无法归类 → 跳过
```

### 截图方式

**只截前台窗口**，不截全屏。通过 `pywin32` 获取前台窗口句柄和窗口区域，只截取该窗口内容。优点：
- 更精准，排除任务栏/其他窗口干扰
- 截图体积更小
- 隐私性更好，不会捕获其他窗口内容

### MVP 截图策略

MVP 使用固定间隔截图（默认 30 秒），不做智能变化检测。后续迭代加入屏幕变化检测和智能频率。

### 活动确认规则

为避免记录用户随手打开又立即关闭的文档，采用"至少两次确认"策略：

```
截图1: Chrome - "某篇文章"    → 标记为"待确认"活动，暂不写入记忆
截图2: Chrome - "某篇文章"    → 同一文档连续出现第2次，确认为真实活动，写入记忆
截图3: Chrome - "某篇文章"    → 合并到现有活动（延长时间）

截图1: WPS - "报告.docx"     → 标记为"待确认"
截图2: Chrome - "另一篇文章"  → "报告.docx"未再出现，丢弃待确认记录
```

同时结合用户在场检测：
- 通过 `GetLastInputInfo`（Win32 API）检测鼠标/键盘空闲时间
- 空闲超过 2 分钟 → 暂停记录，判定用户可能离开
- 前台窗口不是文档类应用 → 跳过

### 截图存储策略

截图保存下来，作为活动记忆的视觉证据，用户问"今天干了什么"时可配合截图回顾。

**存储规则：**

- 截图压缩为中等质量 JPEG（约 100-200KB/张），减少空间占用
- 只在应用运行期间截图，应用关闭即停止
- 截图与活动记忆文档关联，记忆文档中记录截图路径

**同质化清理：**

同一活动期间的多张截图往往内容高度相似（比如持续阅读同一文章），需要去重：

```
截图1: Chrome - "LangGraph教程" (10:00:00)  ──┐
截图2: Chrome - "LangGraph教程" (10:00:30)   │ 相似度>90%，只保留首尾
截图3: Chrome - "LangGraph教程" (10:01:00)   │ + 内容变化较大的帧
截图4: Chrome - "LangGraph教程" (10:01:30)  ──┘

清理后保留：截图1(开始)、截图4(结束)
    如果中间某帧内容变化较大（如翻到文章新章节），也保留
```

- 对比相邻截图的像素差异（结构相似度 SSIM）
- 相似度超过阈值（如 90%）的中间帧删除，只保留首帧、尾帧和内容变化显著的帧
- 无法归类为任何场景的截图直接丢弃，不存储

**过期清理：**

- 默认保留 7 天截图，超期自动清理（可配置）
- 活动记忆文档（文字摘要）永久保留，不受清理影响

**记忆文档中的截图引用：**

```markdown
时间：2026-06-25 10:00-10:35
场景：阅读
来源：Chrome
标签：学习、阅读
截图：screenshots/2026-06-25/10-00-00.jpg, 10-01-30.jpg, 10-35-00.jpg

用户在Chrome中阅读了一篇关于LangGraph工作流编排的技术文章...
```

**存储目录：**

```
data/pet-memory/
├── index.md
├── activities/
│   └── 2026-06-25/
│       └── activity_001.md
└── screenshots/                  截图存储
    └── 2026-06-25/
        ├── 10-00-00.jpg          活动开始帧
        ├── 10-01-30.jpg          内容变化帧
        └── 10-35-00.jpg          活动结束帧
```

### 场景识别

通过 OCR 文本 + 窗口标题（Win32 API pywin32）综合判断：

| 场景 | 识别方式 | 处理方式 |
|------|---------|---------|
| **阅读文章/文档** | 浏览器/阅读器/文档软件 + 大段文字内容 + 窗口标题 | 提取标题和要点，摘要为"阅读了关于XX的文章" |
| **编写文档** | Office/WPS/Notion 等 + 文字编辑特征 + 窗口标题 | 提取文档标题和大纲，摘要为"编写了XX文档" |
| **其他** | 无法归类为以上两种 | 跳过，不记录 |

### 活动合并

连续多次截图如果识别为同一活动（同一文档标题 + 同一场景类型），不重复写入，而是更新现有活动记录的持续时长和内容摘要。检测到活动切换时，关闭上一条记录并创建新记录。

```
截图1: Chrome - "LangGraph教程"  ──┐
截图2: Chrome - "LangGraph教程"   │──► 合并为一条活动
截图3: Chrome - "LangGraph教程"  ──┘
截图4: WPS - "需求文档.docx"     ──► 新活动
```

## 6. 记忆系统

### 设计原则

- 所有记忆以**文档文件**（Markdown）形式存储，没有数据库表
- 记忆内容为自然语言描述，由 LLM 直接阅读和理解
- 活动记忆以独立文档目录管理，不依赖 Hermes 内置 Memory 的存储格式，但通过自定义 Tool 让 Hermes Agent 可以检索这些文档

### 三层记忆结构

类似人类记忆——当天的事记得细节，上周的只记得大概，上个月只有印象：

```
┌─────────────────────────────────────┐
│  第三层：周/月总结                    │  "6月主要学了LangGraph、RAG优化，
│  (定期自动生成，极度浓缩)             │   工作上完成了PRD和API文档编写"
├─────────────────────────────────────┤
│  第二层：每日总结                    │  "今天上午阅读了2篇技术文章，
│  (每天应用关闭时自动生成)             │   下午编写了PRD文档"
├─────────────────────────────────────┤
│  第一层：单条活动记录                │  "[10:00-10:35] 阅读LangGraph教程..."
│  (被动监控实时写入)                  │
└─────────────────────────────────────┘
```

### 目录结构

索引按日期拆分，避免单个文件过大：

```
data/pet-memory/
├── index/                           每日活动索引（第一层入口）
│   ├── 2026-06-25.md
│   ├── 2026-06-24.md
│   └── ...
│
├── activities/                      完整活动记录（第一层详情）
│   └── 2026-06-25/
│       ├── activity_001.md
│       ├── activity_002.md
│       └── ...
│
├── summaries/                       聚合总结
│   ├── daily/                       每日总结（第二层）
│   │   ├── 2026-06-25.md
│   │   └── ...
│   ├── weekly/                      周总结（第三层）
│   │   └── 2026-W26.md
│   └── monthly/                     月总结（第三层）
│       └── 2026-06.md
│
├── screenshots/                     截图存储
│   └── 2026-06-25/
│       ├── 10-00-00.jpg
│       └── ...
│
└── embeddings/                      向量缓存
    └── index.npy
```


### 每日总结示例（summaries/daily/2026-06-25.md）

```markdown
# 2026-06-25 每日总结

- 上午阅读了 LangGraph 工作流编排教程，了解了 StateGraph 节点定义和条件路由
```

### 周/月总结示例（summaries/weekly/2026-W26.md）

```markdown
# 2026年第26周总结 (06-23 ~ 06-29)

## 学习方向
- LangGraph 工作流编排（Agent 框架方向）
- RAG 检索优化
- Python 异步编程

## 工作产出
- 完成"桌面宠物Agent"PRD 文档
- 编写 API 接口设计文档

- 学习阅读：约5小时
- 文档写作：约3小时
```

### 总结生成时机

| 层级 | 生成时机 | 生成方式 |
|------|---------|---------|
| 每日总结 | 应用关闭时 / 当天最后一次活动结束后 | 本地 LLM 读取当日所有活动索引，生成总结 |
| 周总结 | 每周日应用关闭时 / 新一周首次启动时 | 本地 LLM 读取本周所有每日总结，生成周总结 |
| 月总结 | 每月最后一天 / 新月份首次启动时 | 本地 LLM 读取本月所有周总结，生成月总结 |

### 分层检索策略

用户问题的时间跨度不同，检索的记忆层级也不同：

```
用户问："刚才我在看什么？"
      → 第一层：读取今日索引中最近的几条活动记录

用户问："今天做了什么？"
      → 第二层：读取今日总结
      → 如果用户追问细节 → 第一层：加载具体活动文档

用户问："这周学了什么？"
      → 第三层：读取本周周总结
      → 如果用户追问某天 → 第二层：加载那天的每日总结

用户问："上个月的工作内容？"
      → 第三层：读取上月月总结
```

Agent 通过 Tool 拿到对应层级的文档后，由 LLM 自行判断是否需要加载更细的层级。

### Embedding 向量检索

MVP 使用本地 embedding 模型（bge-small-zh），对每日索引条目和各层总结做向量化。当用户提问模糊（如"之前看过一篇关于向量数据库的文章"）时，通过向量相似度跨日期检索。向量缓存为 numpy 文件存储在本地。

## 7. Agent Tools

### 新增工具定义

以下工具通过本地 MCP Server 暴露给 Hermes Agent，Agent 在对话中可自主调用：

**read_activity_index**

```
描述：读取某天的活动索引，获取当天所有活动的一句话摘要列表
输入：
  - date: 日期，如"2026-06-25"、"today"
返回：该天索引文档内容（纯文本）
```

**read_activity_detail**

```
描述：读取某条活动的完整记忆文档
输入：
  - file_path: 记忆文档路径，如"activities/2026-06-25/activity_001.md"
返回：完整记忆文档内容（纯文本）
```

**read_summary**

```
描述：读取每日/周/月总结文档
输入：
  - type: "daily" | "weekly" | "monthly"
  - key: 对应标识，如"2026-06-25"、"2026-W26"、"2026-06"
返回：总结文档内容（纯文本）
```

**list_available_dates**

```
描述：列出有活动记录的日期列表
输入：
  - range: 可选，如"this_week"、"this_month"、"2026-06"
返回：日期列表和每天的活动条数
```

**screenshot_tool**

```
描述：截取当前前台窗口截图
输入：无
返回：截图文件路径
```

**ocr_tool**

```
描述：对图片进行 OCR 文字识别
输入：
  - image_path: 图片文件路径
返回：识别出的文字内容
```

### 被动监控内部调用

被动监控 Loop 直接调用截图和 OCR 的底层函数，不走 Agent Tool 调度，避免不必要的开销。但同时将这些能力注册为 MCP Tool，让 Agent 在需要时也能主动调用。

## 8. LLM 路由

| 场景 | 模型 | 原因 |
|------|------|------|
| 被动监控：OCR 后生成活动摘要 | 本地 Ollama（Qwen2.5-7B） | 频繁调用，需省成本，摘要任务简单 |
| 被动监控：活动分类打标签 | 本地 Ollama（Qwen2.5-7B） | 分类任务简单，本地可胜任 |
| 主动对话：用户问答 | 云端 API（通过 Hermes 配置） | 需要强理解能力 |
| 主动对话：记忆检索判断 | 云端 API | 随对话一起处理 |

通过 Hermes 的 model 配置切换云端模型，本地模型通过 Ollama 管理。

## 10. 独立应用架构（与 Hermes 对接）

### 设计理念

本项目是一个**独立运行的桌面应用**，不是 Hermes 的内部 plugin 或 skill。类似 CC Switch 的定位——独立应用，与 Hermes 对接。

启动时自动拉起 Hermes Agent 作为后台服务进程，通过 HTTP/WebSocket 与之通信，使用 Hermes 的 Agent 能力（对话、Tools、Skills、MCP、Memory）。

### 架构总览

### 通信方式

利用 Hermes 的 gateway 机制。Hermes 原生支持通过 gateway 对外提供 API 服务（供 Telegram/Discord/Slack 等平台接入），我们的桌面应用以同样的方式接入：

```
桌面宠物应用 ←HTTP/WS→ Hermes Gateway (localhost) ←→ Agent Core
```

### 启动流程

```
1. 用户启动桌面宠物应用
      │
      ▼
2. 检查 Hermes 是否已安装
   ├─ 未安装 → 提示用户安装 Hermes
   └─ 已安装 → 继续
      │
      ▼
3. 启动 Hermes gateway 后台服务进程
   (hermes gateway start --headless)
      │
      ▼
4. 等待 gateway 就绪（健康检查）
      │
      ▼
5. 启动被动监控线程
      │
      ▼
6. 显示 PyQt6 对话窗口
      │
      ▼
7. 用户可以开始对话 / 后台持续监控活动
```

### 被动监控与 Hermes 的交互

被动监控模块在本地完成截图 → OCR → 场景分类 → 摘要生成后，通过 Hermes 的 Memory API 将活动记录写入 Hermes 的记忆系统：

```python
# 被动监控写入活动记忆
hermes_client.send_message(
    "请将以下活动记录保存到记忆中：\n"
    "[2026-06-25 10:00-10:35] #学习 #阅读\n"
    "用户在Chrome中阅读了LangGraph教程..."
)

# 或直接调用 Hermes Memory Tool
hermes_client.call_tool("memory", {
    "action": "save",
    "content": "..."
})
```

用户对话时，消息通过通信层发送给 Hermes，Hermes 自动检索记忆并生成回答。

### 对话流转

对话窗口在宠物 Agent 界面中，用户消息由宠物应用下发给 Hermes 处理：

```
用户在宠物对话窗口输入
      │
      ▼
宠物应用通过 HTTP/WS 发给 Hermes Gateway
      │
      ▼
Hermes Agent 处理（检索记忆、调用工具等）
      │
      ▼
Hermes 返回回复 → 宠物应用
      │
      ▼
宠物对话窗口展示回复
```

### Hermes 不可用时的处理

```
宠物应用定期健康检查 Hermes Gateway
      │
      ├─ 正常 → 一切照常
      │
      └─ 不可用（启动失败/进程崩溃/无响应）
            │
            ▼
      ┌─────────────────────────────────────┐
      │  1. UI 显示提醒："Agent 服务异常，    │
      │     对话功能暂时不可用"               │
      │  2. 被动监控继续运行（不依赖 Hermes） │
      │     截图 + OCR + 本地 LLM 摘要      │
      │     活动记录写入本地文档              │
      │  3. 自动尝试重启 Hermes（间隔递增）   │
      │  4. 恢复后 UI 提醒："Agent 服务已恢复" │
      └─────────────────────────────────────┘
```

被动监控不依赖 Hermes——截图、OCR、本地 LLM 摘要都在宠物应用进程内完成，Hermes 挂了也能继续记录活动。只有对话问答依赖 Hermes。

### 自定义 Tools 注册

我们的截图、OCR、活动检索等工具，通过 Hermes 的 MCP 协议暴露给 Agent：

```
桌面宠物应用内启动一个本地 MCP Server
    │
    ▼
Hermes 通过 MCP 连接到这个 Server
    │
    ▼
Agent 可调用我们的自定义工具：
├─ screenshot    截图
├─ ocr_recognize OCR 识别
├─ read_activity_index  读取活动索引
└─ read_activity_detail 读取活动详情
```

这样 Agent 在对话中也能主动调用截图/OCR 等能力。

## 11. 数据流总览

### 被动监控流

```
[定时器 30s]
    │
    ▼
[检查用户空闲时间 (GetLastInputInfo)]
    ├─ 空闲 > 2分钟 → 跳过本次
    │
    ▼
[pywin32 获取前台窗口句柄 + 标题]
    ├─ 非文档类应用 → 跳过本次
    │
    ▼
[截取前台窗口截图 (pywin32 窗口区域)]
    │
    ▼
[PaddleOCR 识别文字]
    │
    ▼
[场景分类器：阅读 or 写作 or 跳过]
    ├─ 无法归类 → 丢弃截图，跳过
    │
    ▼
[活动确认：与上次截图是否同一活动？]
    ├─ 首次出现 → 标记"待确认"，暂不写入
    ├─ 连续第2次出现 → 确认为真实活动
    │
    ▼
[本地 LLM 生成一句话摘要 + 完整记忆]
    │
    ▼
[活动合并判断：是否与已有活动相同？]
    ├─ 是 → 更新现有记忆（延长时间、补充内容）
    └─ 否 → 创建新记忆文档、更新当日索引、
            保存截图（SSIM 去重）、更新 embedding
```

### 主动对话流

```
[用户在宠物对话窗口输入问题]
    │
    ▼
[宠物应用下发给 Hermes Gateway]
    │
    ▼
[Hermes Agent 判断是否需要检索活动记忆]
    │
    ├── 不需要 → 正常回答 / 调用其他 Tool / MCP
    │
    └── 需要 → 根据时间跨度选择检索层级：
        │
        ├── "刚才" → 调用 read_activity_index(today)
        │            读取最近几条活动
        │
        ├── "今天" → 调用 read_summary(daily, today)
        │            读取今日总结，按需加载活动详情
        │
        ├── "这周" → 调用 read_summary(weekly, this_week)
        │            读取周总结
        │
        └── "上个月" → 调用 read_summary(monthly, last_month)
                       读取月总结
    │
    ▼
[LLM 综合记忆文档生成回答]
    │
    ▼
[Hermes 返回回复 → 宠物对话窗口展示]
```

## 12. 配置文件

```yaml
# config.yaml

monitor:
  enabled: true
  interval_seconds: 30           # 截图间隔
  scenes:                        # 启用的场景
    - reading
    - writing
  idle_timeout_seconds: 120      # 空闲超过2分钟暂停监控
  confirm_count: 2               # 同一文档至少出现N次才记录

screenshots:
  quality: 75                    # JPEG 压缩质量 (1-100)
  cleanup_similarity: 0.9        # SSIM 相似度阈值，超过则删除中间帧
  retention_days: 7              # 截图保留天数
  
ocr:
  engine: paddleocr
  lang: ch                       # 中文

llm:
  local:
    provider: ollama
    model: qwen2.5:7b
    base_url: http://localhost:11434
  cloud:
    # 通过 Hermes 配置管理

memory:
  base_dir: ./data/pet-memory
  embedding_model: bge-small-zh

ui:
  window_width: 400
  window_height: 600
```
